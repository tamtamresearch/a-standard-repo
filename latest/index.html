Abstract

This specification defines an example standard for demonstrating versioned documentation management using ReSpec and GitHub Pages. It provides a proof of concept for maintaining multiple versions of standards documentation in a clean, automated way.

This is a fictional standard created to demonstrate repository organization patterns for standards documentation.

Status of This Document

This document is a draft of a potential specification. It has no official standing of any kind and does not represent the support or consensus of any standards organization.

This is a proof of concept specification. It demonstrates how to manage versioned standards documentation using modern tooling.

This document is not affiliated with any standards body and serves purely as a technical demonstration.

Comments and feedback are welcome via GitHub Issues.

1. Introduction

<p>
    Standards documentation needs to evolve over time. As standards mature,
    new versions are published with improvements, clarifications, and new features.
    However, older versions must remain accessible for users who cannot immediately
    upgrade.
</p>

<p>
    This specification demonstrates a clean approach to managing multiple versions
    of standards documentation:
</p>

<ul>
    <li>Source files separated from build artifacts</li>
    <li>Automated deployment via GitHub Actions</li>
    <li>Multiple versions accessible simultaneously</li>
    <li>Framework-agnostic repository pattern</li>
</ul>

<section id="motivation">
    <h3>Motivation</h3>
    <p>
        Many standards projects struggle with version management:
    </p>
    <ul>
        <li>Large repositories full of build artifacts</li>
        <li>Manual deployment steps prone to errors</li>
        <li>Unclear which files are source vs. generated</li>
        <li>Difficulty accessing older versions</li>
    </ul>
    <p>
        This proof of concept addresses these challenges with a clean,
        automated approach.
    </p>
</section>

<section id="scope">
    <h3>Scope</h3>
    <p>
        This specification covers:
    </p>
    <ul>
        <li>Repository organization patterns</li>
        <li>Build automation using ReSpec</li>
        <li>Version deployment strategies</li>
        <li>GitHub Pages integration</li>
    </ul>
    
    <p>
        This specification does <strong>not</strong> cover:
    </p>
    <ul>
        <li>Specific domain vocabularies</li>
        <li>Complex RDF modeling</li>
        <li>Internationalization</li>
        <li>Production deployment at scale</li>
    </ul>
</section>

<section id="audience">
    <h3>Intended Audience</h3>
    <p>
        This document is intended for:
    </p>
    <ul>
        <li>Standards editors and maintainers</li>
        <li>Technical documentation teams</li>
        <li>Open source project maintainers</li>
        <li>Anyone managing versioned technical documentation</li>
    </ul>
</section>

3. Terminology

<p>
    The following terms are used throughout this specification:
</p>

<dl>
    <dt><dfn data-lt="source file|source files">Source File</dfn></dt>
    <dd>
        A human-authored file that serves as input to the build process.
        Source files are maintained in the repository's main branch.
    </dd>
    
    <dt><dfn data-lt="build artifact|build artifacts">Build Artifact</dfn></dt>
    <dd>
        A generated file produced by the build process. Build artifacts
        are deployed to the gh-pages branch but never committed to the
        main branch.
    </dd>
    
    <dt><dfn>Version</dfn></dt>
    <dd>
        A specific release of the standard, identified by a semantic
        version number (e.g., 1.0.0, 1.1.0, 2.0.0).
    </dd>
    
    <dt><dfn>Latest</dfn></dt>
    <dd>
        An alias that always points to the most recent published version
        of the standard.
    </dd>
    
    <dt><dfn data-lt="gh-pages">GitHub Pages Branch</dfn></dt>
    <dd>
        A special Git branch used for hosting static websites via GitHub Pages.
        In this system, it contains only build artifacts.
    </dd>
</dl>

4. Repository Architecture

<p>
    The repository follows a <strong>source-build separation</strong> pattern
    with automated deployment.
</p>

<section id="branch-structure">
    <h3>Branch Structure</h3>
    
    <p>
        The repository uses two branches:
    </p>
    
    <dl>
        <dt><code>main</code> branch (default)</dt>
        <dd>
            Contains source files, build scripts, and configuration.
            This is where development happens.
        </dd>
        
        <dt><code>gh-pages</code> branch</dt>
        <dd>
            Contains only generated HTML files organized by version.
            This branch is managed exclusively by GitHub Actions.
        </dd>
    </dl>
</section>

<section id="directory-structure">
    <h3>Directory Structure</h3>
    
    <pre class="example" title="Main Branch Structure">

standard-repo/ ├── source/ # ReSpec source files │ ├── index.html │ └── config.js ├── scripts/ │ ├── build-spec.js # Build automation │ └── deploy-version.py # Deployment script ├── .github/workflows/ │ └── deploy.yml # CI/CD automation ├── .mise.toml # Tool versions └── package.json # Dependencies 

    <pre class="example" title="gh-pages Branch Structure">

gh-pages/ ├── index.html # Version selector ├── v1.0/ │ └── index.html ├── v2.0/ │ └── index.html └── latest/ └── index.html

5. Build Process

<p>
    The build process converts ReSpec source files into static HTML:
</p>

<ol>
    <li>ReSpec source is loaded in a headless browser (Puppeteer)</li>
    <li>ReSpec JavaScript processes the document</li>
    <li>Processed HTML is extracted</li>
    <li>Static HTML is written to <code>build/index.html</code></li>
</ol>

<pre class="example" title="Building Locally">

5.1 Build the specification

npm run build

5.2 Output: build/index.html

</pre>

6. Deployment Process

<p>
    Deployment is triggered by Git tags and fully automated:
</p>

<ol>
    <li>Developer creates a Git tag (e.g., <code>v1.0.0</code>)</li>
    <li>GitHub Actions detects the tag</li>
    <li>Build script generates static HTML</li>
    <li>Deployment script copies to gh-pages branch</li>
    <li>Version selector is updated</li>
    <li>Changes are pushed to gh-pages</li>
    <li>GitHub Pages publishes the update</li>
</ol>

<pre class="example" title="Deploying a Version">

6.1 Tag a version

git tag -a v1.0.0 -m "Release v1.0.0"

6.2 Push the tag

git push origin --tags

6.3 GitHub Actions handles the rest automatically!

</pre>

7. Versioning Strategy

<p>
    This specification uses <a href="https://semver.org/">Semantic Versioning</a>:
</p>

<ul>
    <li><strong>MAJOR</strong>: Breaking changes</li>
    <li><strong>MINOR</strong>: New features (backwards compatible)</li>
    <li><strong>PATCH</strong>: Bug fixes</li>
</ul>

<p>
    Examples: <code>1.0.0</code>, <code>1.1.0</code>, <code>2.0.0</code>
</p>

<section id="version-immutability">
    <h3>Version Immutability</h3>
    
    <p class="note">
        Once published, a version <strong>MUST NOT</strong> be modified.
        Corrections must be published as a new version.
    </p>
    
    <p>
        This ensures persistent URIs remain stable and citations remain valid.
    </p>
</section>

9. Benefits of This Approach

<ul>
    <li><strong>Clean Repository</strong>: Only source files in main branch (~5MB vs 500MB+)</li>
    <li><strong>Zero Manual Steps</strong>: Tag and push, automation handles the rest</li>
    <li><strong>Multiple Versions</strong>: All versions accessible simultaneously</li>
    <li><strong>Framework Agnostic</strong>: Easy to swap ReSpec for other tools</li>
    <li><strong>PR Friendly</strong>: Diffs show only source changes</li>
    <li><strong>Reliable</strong>: Automated = consistent, error-free deployments</li>
    <li><strong>Fast Clones</strong>: Small repository size means quick setup</li>
    <li><strong>Clear History</strong>: Git log shows only meaningful changes</li>
</ul>

<section id="framework-flexibility">
    <h3>Framework Flexibility</h3>
    <p>
        The repository pattern supports multiple documentation frameworks:
    </p>
    <ul>
        <li><strong>ReSpec</strong>: W3C-style specifications (current)</li>
        <li><strong>mdBook</strong>: Rust-based simplicity</li>
        <li><strong>MkDocs</strong>: Python documentation</li>
        <li><strong>Custom PHP</strong>: Domain-specific generation</li>
    </ul>
    <p>
        Swapping frameworks requires only updating the build script.
        The deployment automation remains unchanged.
    </p>
</section>

B. Implementation Notes

<p>
    This section provides guidance for implementers adopting this pattern.
</p>

<section id="getting-started">
    <h3>Getting Started</h3>
    <ol>
        <li>Clone the repository template</li>
        <li>Install tools with <code>mise install</code></li>
        <li>Install dependencies with <code>npm install</code></li>
        <li>Edit source files in <code>source/</code></li>
        <li>Test with <code>npm run build</code></li>
        <li>Deploy with Git tags</li>
    </ol>
</section>

<section id="best-practices">
    <h3>Best Practices</h3>
    <ul>
        <li>Always test builds locally before tagging</li>
        <li>Use semantic versioning for clear version history</li>
        <li>Document changes in each version's changelog</li>
        <li>Keep source files clean and well-organized</li>
        <li>Let automation handle deployment (no manual steps)</li>
    </ul>
</section>
