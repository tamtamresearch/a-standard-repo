This specification defines an example standard for demonstrating versioned documentation management using ReSpec and GitHub Pages. It provides a proof of concept for maintaining multiple versions of standards documentation in a clean, automated way.

This is a fictional standard created to demonstrate repository organization patterns for standards documentation.

This is a proof of concept specification. It demonstrates how to manage versioned standards documentation using modern tooling.

This document is not affiliated with any standards body and serves purely as a technical demonstration.

Comments and feedback are welcome via GitHub Issues.

Introduction

Standards documentation needs to evolve over time. As standards mature, new versions are published with improvements, clarifications, and new features. However, older versions must remain accessible for users who cannot immediately upgrade.

This specification demonstrates a clean approach to managing multiple versions of standards documentation:

  • Source files separated from build artifacts
  • Automated deployment via GitHub Actions
  • Multiple versions accessible simultaneously
  • Framework-agnostic repository pattern

Motivation

Many standards projects struggle with version management:

  • Large repositories full of build artifacts
  • Manual deployment steps prone to errors
  • Unclear which files are source vs. generated
  • Difficulty accessing older versions

This proof of concept addresses these challenges with a clean, automated approach.

Scope

This specification covers:

  • Repository organization patterns
  • Build automation using ReSpec
  • Version deployment strategies
  • GitHub Pages integration

This specification does not cover:

  • Specific domain vocabularies
  • Complex RDF modeling
  • Internationalization
  • Production deployment at scale

Intended Audience

This document is intended for:

  • Standards editors and maintainers
  • Technical documentation teams
  • Open source project maintainers
  • Anyone managing versioned technical documentation

Terminology

The following terms are used throughout this specification:

Source File
A human-authored file that serves as input to the build process. Source files are maintained in the repository's main branch.
Build Artifact
A generated file produced by the build process. Build artifacts are deployed to the gh-pages branch but never committed to the main branch.
Version
A specific release of the standard, identified by a semantic version number (e.g., 1.0.0, 1.1.0, 2.0.0).
Latest
An alias that always points to the most recent published version of the standard.
GitHub Pages Branch
A special Git branch used for hosting static websites via GitHub Pages. In this system, it contains only build artifacts.

Repository Architecture

The repository follows a source-build separation pattern with automated deployment.

Branch Structure

The repository uses two branches:

main branch (default)
Contains source files, build scripts, and configuration. This is where development happens.
gh-pages branch
Contains only generated HTML files organized by version. This branch is managed exclusively by GitHub Actions.

│   └── index.html
└── latest/
    └── index.html

Build Process

The build process converts ReSpec source files into static HTML:

  1. ReSpec source is loaded in a headless browser (Puppeteer)
  2. ReSpec JavaScript processes the document
  3. Processed HTML is extracted
  4. Static HTML is written to build/index.html
# Build the specification
npm run build

# Output: build/index.html

Deployment Process

Deployment is triggered by Git tags and fully automated:

  1. Developer creates a Git tag (e.g., v1.0.0)
  2. GitHub Actions detects the tag
  3. Build script generates static HTML
  4. Deployment script copies to gh-pages branch
  5. Version selector is updated
  6. Changes are pushed to gh-pages
  7. GitHub Pages publishes the update
# Tag a version
git tag -a v1.0.0 -m "Release v1.0.0"

# Push the tag
git push origin --tags

# GitHub Actions handles the rest automatically!

Versioning Strategy

This specification uses Semantic Versioning:

  • MAJOR: Breaking changes
  • MINOR: New features (backwards compatible)
  • PATCH: Bug fixes

Examples: 1.0.0, 1.1.0, 2.0.0

Version Immutability

Once published, a version MUST NOT be modified. Corrections must be published as a new version.

This ensures persistent URIs remain stable and citations remain valid.

Benefits of This Approach

  • Clean Repository: Only source files in main branch (~5MB vs 500MB+)
  • Zero Manual Steps: Tag and push, automation handles the rest
  • Multiple Versions: All versions accessible simultaneously
  • Framework Agnostic: Easy to swap ReSpec for other tools
  • PR Friendly: Diffs show only source changes
  • Reliable: Automated = consistent, error-free deployments
